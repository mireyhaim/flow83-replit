0. Product Overview
Product Name

Flow 83

Product Type

Business Operating System (OS) – MVP

Product Definition

Flow 83 is a lightweight business operating system for mentors, therapists, and method creators.
It enables them to transform existing knowledge into a 7-day digital transformational journey, delivered as a personalized, daily, AI-powered mobile experience — without relying on 1:1 work.

The MVP is designed to prove:

End-to-end journey creation and delivery

A human, non-generic participant experience

Automated selling and delivery of transformation

Mentor flexibility, experimentation, and editing

1. MVP Goals
What the MVP Must Prove

A mentor can easily create a first active journey

Participants experience clear daily value

AI maintains the mentor’s tone, method, and boundaries

Payment → experience → completion works end-to-end with no manual intervention

Explicit Non-Goals

Scale optimization

Community

Marketplace

Mobile app

Advanced analytics

Strict methodological enforcement

2. Target Users
Mentor (Primary User)

Has existing knowledge, content, or a method

Wants to sell a short guided process

Needs flexibility and the ability to iterate

Comfortable working with a fixed template

Participant (End User)

Purchases a single journey

Uses a mobile device

Progresses day by day

Interacts primarily through chat

3. MVP System Architecture (High Level)

Flow 83 MVP consists of four core components only:

Method Layer (Minimal, Editable)

Mentor Setup & Journey Creation

Journey Engine (7-Day Fixed Flow)

Participant Experience + Chat

4. Method Layer – MVP
Purpose

To introduce structured, method-based thinking while remaining simple and flexible.
This layer guides journey generation without enforcing rigid rules.

Method Entity (MVP Schema)
{
  "method_name": "string",
  "purpose": "string",
  "stages": [
    { "name": "string", "description": "string" }
  ],
  "tone_guidelines": "string",
  "boundaries": "string",
  "target_audience": "string"
}

MVP Rules

Every Journey is associated with one Method

Methods are editable at any time

No versioning

No change history

Editing a Method does not affect already generated journeys

Method updates only apply to future journeys or regeneration

10. Data Persistence & State Management (MVP)
Core Principle

All user interactions, chat messages, journey data, and progress states must be persisted in the database.
No critical state may exist only in memory or client-side storage.

Participant Data Stored

Participant profile (ID, journey ID)

Start date

Current active day

Day completion states

Timestamps

Access status

Chat Persistence

Each chat message must be stored:

{
  "participant_id": "uuid",
  "journey_id": "uuid",
  "day_number": 1,
  "role": "user | assistant",
  "message": "string",
  "created_at": "timestamp"
}


Rules:

Full chat history is persisted

Chat history is reused as context

Chat survives refresh, reconnect, or device change

Mentor Data Stored

Generated journeys

Edited journey content
